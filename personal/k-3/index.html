1. Overview

This program is a basic kernel created for the ARM920T processor. It supports some basic system calls, multiple (non-preemptive) concurrent usertasks. With this, you can schedule user programs to run with a priority.

There is now a main menu with which a user can now control which programs are run, one of which is the K3 user program. For testing purposes the user is not limited to executing only one program at a time. Note: some of the test programs are designed to utilize as much of the system resources as possible, such as the forkbomb or spawner scripts.

2. Documentation

2.1 Syscalls

The following syscalls were added to the kernel

AwaitEvent(int event_type)
Blocks the task until the event defined by event_type happens.

The delay syscalls implement ticks which represent a 10ms time period within the kernel.

Delay(time_t ticks)
Blocks the task for a specific number of ticks specified by the ticks argument.
DelayUntil(time_t ticks)
Blocks the task until a specific time specified by the ticks argument

The following extra system calls were also added for convinience.

Wait(tid_t tid)
Blocks the task until the specified task has died.
Halt( )
Halts the system immediately.
Assert( msg, function, line )
Prints an assert message, then halts the system. This means user programs won't get fragmented assertion prints
UnregisterAs(char *name)
Unregister a service from the nameserver. Uses tid to validate the request.
ForceUnregisterAs(char *name)
Forcefully remove a registered service from the nameserver. (Disregard who the service belongs to)
This syscall was included to make repetitive runs of certain user tests possible.

3. Design & Technical Details

3.1 Structures & Algorythms

3.1.1 Ordered Linked List

To store the list of tasks that are waiting on a delay or a delay_until we use an ordered link list. We predefine a number of nodes equal to the amount of tasks descriptors available at one time so that we know every task can be waiting. We decided to use a linear runtime insert so that we could ensure our resulting list will be sorted. We do this so that we can have a faster read because we will have to check to see if anything needs to be replied to every tick while we only insert it once and read only the top items.

3.1.2 Circular Buffer

A Circular buffer has been added, this is for future usage.

It has been created as a circular linked list, with the standard circular buffer functions. With the circular buffer, we can ensure the space gets reused properly. Also you can now use the standard overwriting property of a circular bufer to nsure the array only has N elements, overwritting the oldest one. This was (and can be) used to display things like the latest N sensor updates.

3.1.3 Linked List

We re-used our linked list structure to hold the list of tasks waiting on each specific event.

3.2 Kernel

3.2.1 Hardware Interrupts

The hardware interrupt is very similar to our current context switch. We decided to keep the two context switches in seperate functions for now even though they share some code.

When a hardware interrupt happens:

  1. We jump to the irq_enter function
  2. We store the user state exactly as the swi_enter does
    • Since we use the same code to load, this must be done identically
  3. We save the link register
  4. We store a constant syscall id SYSCALL_ID_IRQ
    • So our kernel knows that it shouldn't do any system call stuff
  5. Reload the kernel as normal

At the end of the activate function we now load the user PC and CPSR at the same time to ensure that the user's stack is properly loaded and saved.

3.3 User Tasks

3.3.1 Clock Server and Notifiers

Our clockserver is a simple forever loop with a switch statement that just receives and handles the request. It blocks waiting user tasks by not replying until their delay has been reached. Our clock server does not keep time.

Our clock server handles the delay events by adding the requested delay to the current time and storing this number along with the tasks tid in an ordered linked list. For delay_until, we simply do the same but store only the inputted ticks. The server then checks through the ordered linked list for tasks to unblock.

Our clock_notifiers is also a simple forever loop that waits on the TIMER3 interrupt. On every interrupt, the notifier will increment it's cur_time variable to keep time. We have it here so that in the case we choose not to send to the clock server every tick, the time count will not be affects. Every loop, it will send the current time to clockserver.

3.4 Optimizations

3.4.1 Task descriptor

In order to save room on the task descriptor struct (for when we lock it to cache) we removed the return value pointer from it. It is now saved directly into the user's stack during sys_handle. We can do this because we have the user's stack pointer and know the offset of the return value pointer.

3.4.2 Memcpy

Memcpy has been modified to copy in batch sizes. This change has positive, but very minor changes to majority of the copy operations since they are generally not large enough to utilize the optimization. But it does dramatically decrease the execution time of the few large copy operations. Especially if the size of the data is a multiple of 32 bytes, since we use at most 8 registers r3-r10 to perform the copy. This optimization will allow us to perform demanding string copy operations when the time comes.

5. Bugs

  1. Currently the hash table used by the nameserver does not handle collisions. We used a fairly large table and a well tested hash function to deter these collisions.

5.1 Quirks

  1. If a task keeps dying and being created, while no other tasks change status, then its task_id is reused, and incremented by TOTAL_TASKS. Thus you see major jumps in task_id values. Note: we have added a tail pointer to our task array, so now the tasks recycle better, but this can still happen in a very rare case

    e.g. with TOTAL_TASKS = 64

    4 >> 68 >> 132 >> 196 …

  2. As a temporary fix, until we get to the IO Server up and running, when returning from hardware interrupts, the task is scheduled back onto the front of it's priority. This means that it schedules using FILO instead of Round Robin. We do this to avoid user tasks getting interrupted during a print and causing the output to be jumbled up.

There are a few flags that you can pass into the preprocessor, which allow you to try a few things. See the Common.mk file in the root directory. There you will find the DEFINES variable, where you can uncomment these flags and compile. You can also play around with the extra compiler flags.

DEBUG

Enables the ASSERTS flag.

This also enables any debug only code, notably the following:

  • Prints an overview of the kernel state after every task run. This includes the syscall request, its arguments, and any return values (which may not make sense if the syscall does not return a value)
ASSERTS
Enables asserts and other things in the assert.h file,
LOCAL
Used to toggle required changes to compile the test-suite for my machine

We can also toggle some optimizations

ENABLE_CACHE
Turns on the caches
OPT_MEMCPY
Turns on the memcpy optimization
OPT_ATTRIBUTES

Toggles the function attibutes which give gcc more information

const: Doesn't read/write to globals

pure: Only reads globals

noret: The function is not expected to return;

naked: GCC will not add any function entry/exit code
